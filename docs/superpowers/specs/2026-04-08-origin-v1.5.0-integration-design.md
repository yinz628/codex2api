# 本地定制项目合入上游 v1.5.0 设计

## 背景

当前仓库是基于 `james-6-23/codex2api` 的本地改造版，现有本地 `main` 已包含：

- 代理管理增强
- 代理质量检测与国家筛选
- 所选代理导出按钮
- Docker 低内存构建优化

当前本地分叉点为上游 `4fd0b98`（`v1.3.0` 附近），而上游 `origin/main` 已推进到 `29462c0`（`v1.5.0`）。自分叉点之后：

- 本地分支新增 7 个提交
- 上游新增 65 个提交

上游变更覆盖范围较广，涉及：

- `proxy/*` 请求转发与兼容性
- `auth/*` 调度与状态管理
- `proxy/wsrelay/*` 会话复用与稳定性
- `admin/*` OAuth、导入、测试连接、后台接口
- `database/*` SQLite/Postgres 容错与持久化
- 前端 Docs/API Reference/Accounts/Settings/Usage 页面

## 目标

在不丢失本地定制功能的前提下，将上游 `origin/main` 的更新完整合入本地项目，并建立新的同步基线。

交付结果应满足：

- 本地功能继续可用
- 上游 `v1.5.0` 的核心后端修复与功能被保留
- 代码库可通过现有主要验证
- 后续继续同步上游的成本降低

## 非目标

本次不包含以下内容：

- 不重写本地定制功能为全新实现
- 不额外设计新的产品需求
- 不在合入过程中顺手做无关重构
- 不主动舍弃上游大块功能，只因页面样式或文案变化不同
- 不默认直接推送远程或覆盖部署环境

## 方案对比

### 方案 A：直接在当前 `main` 上合并上游

优点：

- 路径最短

缺点：

- 冲突处理与验证中途出问题会直接污染主线
- 不利于审查本地定制是否被覆盖

### 方案 B：在独立集成分支上完整合并上游

优点：

- 风险隔离最好
- 适合处理多文件冲突和大范围回归验证
- 便于在合并完成后再决定是否回合到本地 `main`

缺点：

- 步骤稍多

### 方案 C：只挑选稳定性修复进行 cherry-pick

优点：

- 可避开部分上游 UI/Docs 变化

缺点：

- 上游提交数量多且跨模块耦合强，容易漏改
- 后续再次同步上游时成本更高

## 选型

选择 **方案 B：在独立集成分支上完整合并上游**。

理由：

- 最适合当前“本地已有定制 + 上游已大幅演进”的状态
- 能完整保留上游 `v1.5.0` 的价值
- 能在不污染主线的情况下处理冲突和回归
- 后续继续同步时更容易建立稳定基线

## 集成原则

### 1. 上游基线优先

凡是上游针对稳定性、兼容性、缓存、连接复用、SQLite 容错、OAuth 流程、测试修复的变更，默认优先保留。

### 2. 本地定制保留

以下本地能力属于业务定制，默认保留：

- 代理质量检测与测试结果落库
- 代理国家筛选与排序
- 所选代理导出按钮
- Docker 低内存构建优化

### 3. 冲突处理遵循“功能完整优先”

当上游 UI/Docs/API Reference 与本地页面同时修改时：

- 先保证功能完整
- 再保证路由、接口、文案可用
- 样式差异和次级交互差异可接受上游或本地任一实现，只要不影响功能

### 4. 最小修复原则

若合并后出现测试失败或编译失败，只修复由“本地定制与上游新基线不兼容”直接导致的问题，不扩展到无关重构。

## 执行设计

### 阶段 1：建立集成分支

- 从当前本地 `main` 切出集成分支，例如 `codex/merge-origin-v1.5.0`
- 在该分支执行对 `origin/main` 的真实 merge
- 不在此阶段直接修改本地 `main`

### 阶段 2：处理核心共享模块

优先检查高风险共享区域：

- `proxy/*`
- `proxy/wsrelay/*`
- `auth/*`
- `admin/handler.go`
- `admin/oauth.go`
- `database/postgres.go`
- `database/sqlite.go`
- `main.go`

重点确认：

- 上游新引入的会话亲和、连接复用、状态重置逻辑仍存在
- 本地代理测试与质量检测逻辑未被覆盖
- 数据库字段与索引兼容，不丢失本地新增字段

### 阶段 3：处理本地定制功能

重点检查本地差异点：

- `frontend/src/pages/Proxies.tsx`
- `frontend/src/api.ts`
- `admin/proxy_quality.go`
- `admin/proxy_quality_test.go`
- `Dockerfile`
- `docker-compose.sqlite.yml`

确认内容：

- 代理导出按钮仍存在
- 代理质量检查接口与字段映射仍可用
- 国家筛选、质量排序、展示列不被上游页面重构覆盖
- Docker 串行编译优化仍保留

### 阶段 4：回归前端新增页面与路由

上游新增或大改区域需要额外检查：

- `frontend/src/App.tsx`
- `frontend/src/components/Layout.tsx`
- `frontend/src/pages/ApiReference.tsx`
- `frontend/src/pages/Guide.tsx`
- `frontend/src/pages/Accounts.tsx`
- `frontend/src/pages/Settings.tsx`
- `frontend/src/pages/Usage.tsx`

确认内容：

- 新页面能正常构建
- 路由不覆盖本地已有入口
- 本地代理管理页面仍可访问

## 数据与接口兼容性

### 数据库

本次设计假定数据库迁移继续沿用现有“增量向后兼容”的方式，不引入破坏性回滚。

需要特别核对：

- `proxies` 表的本地字段是否仍被保留
- 上游 SQLite 容错与索引变更是否与本地字段共存
- 账号状态、用量缓存、OAuth 相关字段是否正确迁移

### 接口

需重点确认以下接口族不出现行为回退：

- 代理测试与代理管理接口
- 账号导入接口
- OAuth 相关接口
- `/v1/chat/completions`
- `/v1/responses`
- 上游新增兼容路由或消息接口

## 风险与应对

### 风险 1：共享核心文件自动合并成功但语义冲突

应对：

- 不只依赖 Git merge 成功
- 合并后通过定向代码核对和测试验证确认最终语义

### 风险 2：前端 Docs/API Reference 路由与本地页面入口冲突

应对：

- 检查 `App.tsx`、`Layout.tsx`、导航项和路由定义
- 以“保留本地功能入口 + 接纳上游新增页面”为优先目标

### 风险 3：数据库迁移在 SQLite 模式下出现兼容问题

应对：

- 运行现有 `go test ./...`
- 特别关注 `database/*` 与相关 admin/auth 测试

### 风险 4：低内存 Docker 构建优化被覆盖

应对：

- 合并后重新确认 `Dockerfile` 中 `GO_BUILD_PARALLEL`
- 用等价 Linux 交叉编译命令验证

## 验证设计

至少执行以下验证：

### 编译与测试

- `frontend` 下执行 `npm run typecheck`
- `frontend` 下执行 `npm run build`
- 根目录执行 `go test ./...`

### 定向功能回归

- 代理管理页面构建成功
- 代理质量字段在前后端类型上保持一致
- 代理导出功能代码仍存在且无类型错误
- OAuth 相关后端代码可编译
- `wsrelay`、`auth`、`proxy` 相关测试通过

### Docker 构建等价验证

若本地无 Docker：

- 执行 `CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -p=1 -ldflags="-s -w" -o <temp> .`

若环境有 Docker：

- 额外执行一次目标镜像构建验证

## 收尾策略

若集成分支验证通过，再让用户选择：

1. 合回本地 `main`
2. 保留集成分支稍后处理
3. 推送集成分支用于审查

默认不自动推送远程。

## 成功标准

满足以下条件视为本次集成成功：

- 上游 `origin/main` 已合入本地项目
- 本地定制功能未丢失
- `npm run typecheck` 通过
- `npm run build` 通过
- `go test ./...` 通过
- 若涉及 Docker 低内存构建路径，等价 Linux 交叉编译验证通过
