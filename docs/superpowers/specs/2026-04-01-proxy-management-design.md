# 代理管理增强设计

## 背景

当前代理管理页位于 `frontend/src/pages/Proxies.tsx`，已经支持：

- 代理列表展示
- 本地分页（固定每页 10 条）
- 单条删除
- 批量选择后删除
- 代理测试与测试结果展示
- 代理池启用/禁用

本次需求是在不扩大系统边界、不调整数据库 schema 的前提下，为代理管理补足三个可直接提升日常运维效率的能力：

1. 增加每页显示数量选择：`10 / 50 / 100 / 200`
2. 完善批量删除按钮与删除流程
3. 增加按地区筛选代理的功能

## 目标

在保持现有代理管理页交互风格和后端接口兼容的前提下，让用户可以：

- 按自己需要查看更多或更少的代理条目
- 更安全、更清晰地执行批量删除
- 基于当前已测试得到的地区信息快速筛选代理

## 非目标

本次不包含以下内容：

- 不将代理列表改为后端分页
- 不新增数据库字段保存独立的 `country / region / city`
- 不修改现有代理测试接口的返回结构
- 不重构代理页为多个子组件
- 不新增复杂筛选组合（如延迟区间、启用状态、关键词搜索）

## 现状与约束

### 当前实现

- 代理页通过 `api.listProxies()` 一次性拉取全部代理
- 页面使用固定常量 `PAGE_SIZE = 10`
- 当前分页发生在前端，逻辑为 `proxies.slice(...)`
- 批量删除调用 `/api/admin/proxies/batch-delete`
- 地区展示使用 `ProxyRow.test_location`
- `test_location` 是代理测试后回写的拼接字符串，不是结构化字段

### 约束

- 需要尽量走轻量改动路径
- 不能引入数据库迁移或接口重构
- 需要兼容已有单条删除、代理测试、一键测试和启用/禁用能力
- 文案需要同步到中英文语言文件

## 方案对比

### 方案 A：前端增强为主

做法：

- 保持一次性拉取全部代理
- 在前端完成地区筛选、每页数量切换和分页
- 在前端完善批量删除交互与状态管理

优点：

- 改动最小
- 不改后端接口与数据库
- 交付速度快，风险低

缺点：

- 依赖 `test_location` 文本作为地区来源
- 当代理数量极大时，本地分页的性能上限低于服务端分页

### 方案 B：前后端一起升级为服务端分页筛选

做法：

- 扩展 `GET /api/admin/proxies` 支持 `page`、`page_size`、`region`
- 前端切换为服务端分页和筛选

优点：

- 更适合超大规模代理列表
- 接口语义更完整

缺点：

- 后端改动更大
- 需要补更多接口测试
- 超出当前需求最短路径

### 方案 C：结构化持久化地区信息

做法：

- 代理测试结果拆分为独立 `country / region / city`
- 以结构化字段支持筛选和统计

优点：

- 后续扩展能力最强

缺点：

- 需要改表结构和持久化逻辑
- 明显超出本次需求边界

### 选型

选择 **方案 A：前端增强为主**。

理由：

- 完整覆盖本次三个功能点
- 不引入 schema 变更
- 与当前页面的“前端本地分页”模式一致
- 最符合“满足质量要求的最短路径”

## 详细设计

### 1. 每页显示数量选择

#### 交互

- 在代理表格区域增加每页数量选择器
- 固定提供四个选项：`10 / 50 / 100 / 200`
- 默认值仍为 `10`
- 当用户切换每页数量时，当前页重置为第 `1` 页

#### 实现

- 将 `PAGE_SIZE = 10` 改为页面状态 `pageSize`
- 将 `totalPages` 和 `pagedProxies` 改为基于筛选结果与 `pageSize` 计算
- 保持分页按钮样式和现有视觉风格一致

#### 预期结果

- 用户可快速切换列表密度
- 不会因切换页容量导致当前页越界或空页

### 2. 地区筛选

#### 交互

- 在代理页增加地区下拉筛选
- 选项来源于当前列表中的 `test_location` 去重结果
- 固定增加两个特殊选项：
  - `全部地区`
  - `未测试`

#### 过滤规则

- `全部地区`：显示所有代理
- `未测试`：显示 `test_location` 为空字符串的代理
- 其他地区选项：按完整 `test_location` 精确匹配

#### 实现

- 新增页面状态 `regionFilter`
- 基于 `proxies` 计算 `regionOptions`
- 先做地区过滤，再做分页切片
- 切换地区后，当前页重置为第 `1` 页

#### 预期结果

- 用户可快速按地区缩小列表
- 不需要额外后端接口和数据库改造

### 3. 批量删除完善

#### 当前问题

- 批量删除没有确认步骤
- 删除过程没有 loading / disabled 状态
- 选中项在数据刷新后仅靠手工清空，交互不够稳
- 顶部按钮仅在有选中项时出现，反馈较弱

#### 交互改进

- 批量删除按钮继续保留在顶部操作区
- 有选中项时显示数量
- 点击后弹出确认框，明确本次删除数量
- 删除进行中按钮禁用并显示进行中状态
- 删除成功后清空已删除项的选中状态
- 删除完成后自动修正当前页，避免落在空页

#### 选择语义

- 选择集合仍然以全局 `selected` 为准，不仅限当前页
- 当前页“全选”只影响当前页可见项
- 批量删除会删除所有当前已选中的代理，即使这些代理跨页存在

#### 实现

- 新增页面状态 `batchDeleting`
- 在调用 `api.batchDeleteProxies([...selected])` 前执行确认
- 删除成功后：
  - 清空 `selected`
  - 重新拉取列表
  - 由页码修正逻辑自动回到有效页
- 如果列表因筛选变化而减少，继续复用已有页码修正逻辑

#### 预期结果

- 降低误删风险
- 明确反馈删除状态
- 让跨页勾选与删除行为更可预期

## 数据流

### 页面加载

1. `reload()` 并行获取代理列表和系统设置
2. 代理列表写入 `proxies`
3. 系统设置中的 `proxy_pool_enabled` 写入 `poolEnabled`
4. 页面基于 `proxies` 计算：
   - `enabledCount`
   - `regionOptions`
   - `filteredProxies`
   - `totalPages`
   - `pagedProxies`

### 地区筛选与分页

1. 用户切换地区
2. 更新 `regionFilter`
3. 页码重置为 `1`
4. 基于 `filteredProxies` 重新计算分页结果

### 每页数量切换

1. 用户切换每页数量
2. 更新 `pageSize`
3. 页码重置为 `1`
4. 基于新页容量重算 `totalPages` 和 `pagedProxies`

### 批量删除

1. 用户勾选代理
2. `selected` 维护全局已选 ID 集合
3. 用户点击批量删除
4. 页面先进行确认
5. 确认后调用现有批量删除接口
6. 删除成功后清空选择并执行 `reload()`
7. 页面根据新数据自动修正页码

## 错误处理

- 地区筛选和每页数量切换均为纯前端状态变化，不额外引入失败分支
- 批量删除失败时：
  - 保留当前选择，不自动清空
  - 结束 loading 状态
  - 保持当前页与筛选条件不变
- 单条删除、测试代理、一键测试、启用/禁用逻辑保持原有行为

## 测试与验证

### 定向功能验证

1. 默认每页显示 10 条
2. 切换到 50 / 100 / 200 后，列表条数和分页数量正确
3. 选择某个地区后，仅显示该地区代理
4. 选择“未测试”后，仅显示 `test_location` 为空的代理
5. 切换筛选条件后页码自动回到第 1 页
6. 当前页全选仅影响当前页显示的代理
7. 跨页勾选后，批量删除会删除所有已选代理
8. 批量删除前会出现确认提示
9. 批量删除中按钮禁用，完成后选中状态清空
10. 删除后若当前页为空，页码会自动落回有效页

### 回归验证

1. 单条删除仍可用
2. 启用/禁用代理仍可用
3. 单条测试仍可用
4. 一键测试仍可用
5. 代理池开关仍可用

## 风险与权衡

- 地区筛选依赖 `test_location` 文本，不适合做模糊拆分或多层地区筛选
- 未测试代理只有在执行过代理测试后才会获得地区信息，因此“地区筛选”天然是增量能力
- 由于仍然是前端本地分页，当代理规模未来继续扩大时，可能需要升级到服务端分页方案

## 后续演进建议

若未来代理规模继续增长或需要更丰富筛选能力，可作为下一阶段单独设计：

- 为代理列表增加后端分页和筛选参数
- 结构化持久化 `country / region / city`
- 增加按启用状态、标签、延迟范围的组合筛选
